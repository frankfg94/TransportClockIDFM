# Transport Clock GPT

Nuxt 3 + Vue 3 + TypeScript application showing IDFM PRIM departures and dynamic line diagrams.

- IDFM PRIM data powers the next departure boards.
- NeTEx data powers dynamic line diagrams.
- The UI is internationalized in French and English.

Android APK distribution is documented in [docs/mobile-release.md](docs/mobile-release.md).
The i18n workflow is documented in [docs/i18n.md](docs/i18n.md).

Secure GTFS line geometry and import are documented in [docs/gtfs-line-geometry.md](docs/gtfs-line-geometry.md).

## Run the Project

```powershell
npm.cmd install
npm.cmd run dev
```

Then open `http://localhost:3000` or the URL printed by Nuxt.

The PRIM key is read from `.env.local` through `IDFM_API_KEY`. Calls to `"/api/idfm"` go through the Nitro route `server/api/idfm/[...path].ts`, so the browser never receives the key.

## Internationalization

Every user-facing string must go through `src/i18n` instead of using raw text in components.

```vue
<script setup lang="ts">
import { useI18n } from "../src/i18n";

const { t } = useI18n();
</script>

<template>
  <button>{{ t("common.actions.save") }}</button>
</template>
```

When adding a new string, add the same key to `src/i18n/messages/fr.ts` and `src/i18n/messages/en.ts`. TypeScript verifies that both catalogs keep the same structure. The language can be changed from Settings; automatic mode uses French when the browser timezone is `Europe/Paris`, otherwise English.

## Offline NeTEx Cache

The frontend reads the JSON cache generated by `idfm-node-backend`. Locally, it automatically looks for:

```txt
../idfm-node-backend/public/data/netex
```

NeTEx data is not required to boot the app, but a warning will be displayed and line diagrams cannot be generated without this cache.

To update the NeTEx cache:

```powershell
rclone copy . <Your formatted NeTEx cache remote>:idfm-backend-netex-cache/netex/current --progress --transfers 16 --checkers 32 --fast-list
```

You can explicitly configure the NeTEx cache with two separate variables. `IDFM_NETEX_CACHE_REMOTE` has priority and forces a private R2 or HTTP(S) remote cache. `IDFM_NETEX_CACHE_LOCAL` forces a local folder. If neither variable is set, Nuxt automatically searches the local development cache.

```powershell
$env:IDFM_NETEX_CACHE_LOCAL="C:\Users\franc\AndroidStudioProjects\VibeIDFM\idfm-node-backend\public\data\netex"
```

Private R2 example, recommended in production:

```powershell
$env:IDFM_NETEX_CACHE_REMOTE="r2://idfm-backend-netex-cache/netex/current"
$env:R2_ACCOUNT_ID="<cloudflare-account-id>"
$env:R2_ACCESS_KEY_ID="<r2-access-key-id>"
$env:R2_SECRET_ACCESS_KEY="<r2-secret-access-key>"
```

With `r2://`, JSON files stay private: the browser never receives R2 credentials and cannot read the bucket directly. Only the Nuxt Nitro backend signs server-to-server R2 requests.

Public HTTP example, useful only for quick tests:

```powershell
$env:IDFM_NETEX_CACHE_REMOTE="https://pub-xxxxx.r2.dev/netex/current"
```

The Nuxt endpoint `GET /api/lines/:lineId/topology` adapts these JSON files to the UI model. The visual graph consumes the `schematic` contract generated server-side, without calling Navitia to rebuild topology.

Run tests:

```powershell
npm.cmd run test
```

## Capacitor Mobile Version

The Nuxt frontend is embedded as static files; Nitro routes under `/api/*` remain on the Cloudflare deployment. Copy `.env.capacitor.example` to `.env.capacitor` (it is ignored by Git) and fill in the public backend URL:

```txt
CAPACITOR_BUILD=true
NUXT_PUBLIC_API_BASE_URL=https://your-deployment.pages.dev
```

Then generate and sync the Android app:

```powershell
npm.cmd run build:capacitor
npm.cmd run capacitor:sync
npm.cmd run capacitor:android
```

Deploy this Nuxt backend version first: it adds the CORS headers required for calls from the mobile WebView.

The same frontend can later receive the iOS platform on a Mac with `npx cap add ios`.

## Home Assistant API

The Home Assistant integration uses a versioned, stateless server API:

- `GET /api/ha/v1/info`
- `GET /api/ha/v1/catalog/families`
- `GET /api/ha/v1/catalog/lines?family=&q=`
- `GET /api/ha/v1/catalog/stations?family=&lineId=&q=`
- `GET /api/ha/v1/catalog/directions?family=&lineId=&stationId=`
- `POST /api/ha/v1/boards`

The Bearer token is optional. To make it required:

```powershell
$env:TRANSPORT_CLOCK_HA_TOKEN="<token>"
$env:TRANSPORT_CLOCK_INSTANCE_ID="<stable-identifier>"
```

`TRANSPORT_CLOCK_INSTANCE_ID` must remain stable across deployments so Home Assistant keeps recognizing the same instance.

## Add a Line or Stop

Default boards live in `src/config/transitBoards.ts`.

For the add modal, `src/services/boardBuilder.ts` turns a line + station selection into a `TransitBoardConfig`, then preferences are persisted through `src/storage/transitPreferences.ts`.
